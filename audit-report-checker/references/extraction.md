# 提取策略原理

本文件说明审计报告解析层（`scripts/parse_report.py）的四路分流策略、不同提取工具的精度边界，以及数字清洗规则。理解这些原理有助于 Claude 在读取 `report.md` 时正确判断各部分数字的置信度，并在报告中对 OCR 路径的勾稽结果给出"建议人工复核"提示。

**Claude 何时读这个文件**：
- 看到 `report.md` 中带 `SOURCE: mineru` 注释的章节时，理解该部分数字经 OCR 提取，置信度低于文本路径
- 用户问"为什么这个数字勾稽不上"时，判断是否为 OCR 精度问题
- 用户问"为什么要装 mineru"或"能不能只用 pdfplumber"时，解释四路分流的必要性
- 配置环境时（首次遇到扫描件/图片报表），引导用户安装 mineru

---

## 一、四路分流

审计报告格式多样，没有一种工具能通吃。`parse_report.py` 按文档类型和内容特征自动选择最佳提取路径：

```
输入文档
  │
  ├─ Word (.docx)
  │    ├─ 文本表格为主（无图片报表）→ python-docx 直读（精确、快）
  │    └─ 图片报表 / inline_shapes 偏多 → mineru extract --model vlm（云端 OCR 转 markdown）
  │
  └─ PDF
       ├─ 文本型（首页文本 > 50 字）→ pdfplumber（精确，数字 100% 可靠）
       └─ 扫描型（首页几乎无文本）→ mineru extract --model vlm
```

### 1.1 Word 文本表格（python-docx）

- **适用**：报告正文和附注以 Word 表格形式存在的 .docx 文件（如博达新材附注、盛屯附注、CTM 附注）
- **原理**：python-docx 直接读取 XML 中的 `<w:tbl>` 结构，数字是文本节点，100% 精确
- **判断**：python-docx 扫描文档时，若发现财务报表应有的表格缺失，或 `inline_shapes` 数量偏多（疑似报表以图片插入），则改用 mineru

### 1.2 Word 图片报表（mineru）

- **适用**：完整审计报告 Word 版本中，财务报表以图片形式插入（报表格式复杂直接截图）
- **原理**：mineru `extract --model vlm` 把整个文档渲染成图片，用视觉模型识别表格结构并输出 markdown
- **判断**：python-docx 扫描时检测到报表表格缺失或图片过多，自动切换

### 1.3 PDF 文本型（pdfplumber）

- **适用**：原生数字 PDF（从 Word/Excel 导出，非扫描件），如金星化工 PDF、芜湖盐业小贷 PDF
- **原理**：pdfplumber 直接读取 PDF 文本层数字，不经 OCR，100% 精确
- **判断**：首页文本量 > 50 字即认为是文本型 PDF

### 1.4 PDF 扫描型（mineru）

- **适用**：扫描件 PDF（纸质报告扫描成电子版），如部分老报告或外部获取的报告
- **原理**：mineru `extract --model vlm` 把 PDF 每页渲染成图片，用视觉模型 OCR 识别
- **判断**：首页几乎无文本（pdfplumber 提取不到文字）即认为是扫描型

## 二、pdfplumber vs mineru 对照表

文本型 PDF 优先用 pdfplumber，这是审计核查的精度底线。mineru 仅在无文本层时使用。

| 维度 | pdfplumber | mineru extract --model vlm |
|---|---|---|
| 提取方式 | 直读 PDF 文本层 XML | 把 PDF 渲染成图片，视觉模型 OCR |
| 数字精度 | **100% 精确**（文本层就是原始数字） | extract vlm **高精度**（实测勾稽双 MATCH）；⚠️ flash-extract 数字不可靠 |
| 速度 | 快（毫秒级/页） | 慢（云端 API 调用，秒级/页） |
| 成本 | 免费（本地） | 免费 token 有额度限制，超出需付费 |
| 数据隐私 | 本地处理，数据不出机器 | **数据上传 mineru.net 云端**，需用户知情同意 |
| 适用场景 | 文本型 PDF（首选） | 扫描型 PDF、图片报表（无文本层时才用） |

### 2.1 mineru 两种模式：vlm 可靠，flash-extract 不可靠

mineru 有两种提取模式，数字精度差异巨大：

**`extract --model vlm`（推荐，高精度）**：
- 视觉语言模型 OCR，实测数字精确（盛屯图片报表验证：资产总计 43,113,912,217.70 = 流动 22,456,392,080.66 + 非流动 20,657,520,137.04；负债合计 25,368,146,281.50 = 流动负债 + 非流动负债，双 MATCH）
- 千分位逗号、小数点格式正确，勾稽可验证
- 需 token（免费额度），速度慢（秒级/页）
- **图片报表的首选 OCR 方案**

**`flash-extract`（免 token，但数字不可靠）**：
- 千分位逗号变小数点：`1,234,567.89` → `1.234.567.89`，数量级错误 1000 倍
- 数字位数丢失/混淆：`0` 和 `O`、`1` 和 `l` 混淆；小数点后位数丢失
- 仅适合结构预览，**不要用于勾稽数字提取**

parse_report.py 默认用 `extract --model vlm`。token 失效时（如 mineru.net 服务端 bug #5133）会提示用户。flash-extract 不作为勾稽提取路径。

### 2.2 为什么文本型仍优先

即使 vlm 高精度，文本型 PDF/Word 仍优先用 pdfplumber/python-docx，原因：
- 速度：pdfplumber 毫秒级/页，mineru 秒级/页（云端 API）
- 隐私：pdfplumber 本地处理，mineru 上传云端（审计报告敏感，需用户知情同意）
- 成本：pdfplumber 免费，mineru 有 token 额度限制
- 精度：文本层是原始数字，0 误差；vlm 虽高精度但仍建议关键数字人工复核

mineru 路径提取的数字，报告标注"OCR 提取，关键勾稽数字建议人工复核"。

## 三、数字清洗

PDF 提取的数字常有格式污染，`parse_report.py` 在提取后做清洗，calculator 在验算时也做容错解析。

### 3.1 pdfplumber 的空格污染问题

pdfplumber 提取 PDF 文本时，数字中间可能被插入空格（PDF 排版产生的字符间距），例如：
- `1,234,567.89` 被提取为 `1, 234, 567.89` 或 `1 234 567.89`
- `123456789` 被提取为 `123 456 789`

**清洗策略**：
- 识别为金额的字符串，先去除所有空白字符（空格、Tab、不可见字符）
- 再按千分位规则解析（逗号或全角逗号分隔，小数点为分隔符）
- 单元格内容若清洗后仍无法解析为数字，保留原值不强行转换

### 3.2 calculator 支持的数字格式

calculator 工具在验算时自动处理以下格式（详见 `scripts/calculator.py）：
- 千分位逗号：`1,234,567.89` → 1234567.89
- 全角逗号：`1，234，567.89` → 1234567.89
- 括号负数：`(1,234.56)` 或 `（1,234.56）` → -1234.56
- 横线/破折号：`—` 或 `-` → 0（报表中无金额时常用横线表示）
- 全角数字：`１２３４.５６` → 1234.56
- 万元单位：报表表头注明"单位：元/万元/千元"时，calculator 按单位换算后再比较

### 3.3 提取路径标记

`report.md` 中每个章节或表格块开头会有 `SOURCE` 注释，标记该部分的提取路径：

```markdown
<!-- SOURCE: pdfplumber, page=6 -->
## 合并资产负债表
...

<!-- SOURCE: mineru, page=12, confidence=high -->
## 货币资金附注
...
```

**标记的意义**：
- **定位**：Claude 报告问题时可引用页码，方便用户翻原件核对
- **置信度判断**：`SOURCE: pdfplumber` 的数字勾稽差异 = 确定错误（红）；`SOURCE: mineru` 的数字勾稽差异 = 异常（黄），提示人工复核
- **降级提示**：mineru 路径的关键勾稽（如资产=负债+权益）失败时，报告中优先提示"数字经 OCR 提取，请先核对原件数字是否正确"，而非直接报错

## 四、mineru 云端 API 的隐私考量

mineru 是云端 API（详见 `mineru_usage.md`），调用时数据会上传到 mineru.net 服务器。审计报告含敏感财务数据，需注意：

- **知情同意**：首次使用 mineru 路径前，Claude 在终端打印隐私提示，告知用户数据将上传云端，征得同意后再调用
- **优先文本路径**：对保密/未公开报告，优先用 pdfplumber/python-docx 文本路径；只有无文本层时才用 mineru
- **客户合规**：上市公司年报属公开信息，用 mineru 无隐私问题；未公开的审计底稿/客户内部报告，需确认客户合规要求后再用

隐私提示由 `parse_report.py` 在调用 mineru 前自动打印，无需 Claude 手动判断。
