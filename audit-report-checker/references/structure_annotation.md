# AI 结构标注 prompt 模板

本文件是消除横加竖加误报的核心机制。Claude 按此模板对附注表格做结构标注（行类型、列类型、横加关系），代码（`scripts/run_check.py`）按标注做竖加和横加验算。算术由代码执行保证确定性，结构判断由 Claude 完成处理变体无穷的表格格式。

移植自 legacy 工具 `legacy/modules/report_note_checker/checkers/auto_table_checker.py` 的 `_STRUCTURE_PROMPT` 和 `_HORIZONTAL_PROMPT`，由旧工具的独立 OpenAI client 调用改为 Claude 主模型直接执行。

**Claude 何时读这个文件**：
- 执行横加竖加检查时（标准/深度检查包含此项）——按本文件 prompt 模板标注附注表格
- 标注失败需要降级时——查"标注失败降级策略"章节
- 用户问"为什么不会有横加竖加误报"时——解释标注+代码验算的双重机制
- 批量标注提效时——查"批量标注提效建议"章节

---

## 一、为什么需要 AI 结构标注

附注表格格式变体无穷，纯代码规则无法覆盖：

| 变体类型 | 举例 | 纯代码规则的失败 |
|---|---|---|
| 子项标记 | "其中："/"组合1："/"（1）"/"[子项]" | 关键词字典无法穷举，漏掉一个就重复加总 |
| 减项前缀 | "减："/"减:"/"减项" | 正负号判断错误，竖加结果反 |
| 多层嵌套 | 公允价值层次表（第一/二/三层）| level 判断错误，小计被当明细重复加 |
| 多行表头 | PDF 提取的合并单元格拆分后产生空列名 | 列号映射错位，取错列的数 |

**核心思路**：Claude 理解表格语义做结构标注，代码按标注做确定性验算。这样既能处理变体，又保证算术 100% 可靠。

典型误报案例（金星化工样本）：流动负债表里"应付职工薪酬"行下面跟着"其中：社会保险费"、"工伤保险费"等子项。纯代码 `is_detail_row`（基于"其中:"关键词）会把子项也当明细加总，导致流动负债合计虚增 95,198.79。AI 标注把子项标为 `sub_detail` 并在竖加时排除，消除误报。

## 二、标注流程概览

对每张待验算的附注表格，Claude 分两步标注（可合并为一次调用）：

1. **结构标注**：标注每行的 `type`（detail/sub_detail/total/subtotal）、`level`（1/2）、`op`（+/-），每列的 `type`（label/amount/percentage/text）
2. **横加判断**：判断列间是否存在加减勾稽关系（如账面余额−坏账准备=账面价值）

代码侧：
- **竖加**：按行标注的 type/level/op，把同级 detail 行求和，核对是否等于 total 行（代码推断，确定性）
- **横加**：按横加判断的 operands/result_col，逐行验算列间加减（代码验算，确定性）
- **代码兜底修正**：对"其中"/"减："/"加："做强制规则修正（AI 偶尔标错，代码兜底）

## 三、行标注规则

### 3.1 type 取值

| type | 含义 | 举例 |
|---|---|---|
| `detail` | 明细数据行，参与合计 | "货币资金"、"银行存款" |
| `sub_detail` | 子项行，属于上一个 detail 行的明细拆分，**竖加时排除** | "其中：医疗保险费"、"组合1：" |
| `total` | 合计/总计行 | "资产总计"、"流动资产合计" |
| `subtotal` | 小计行（标签必须含"小计"二字），竖加时按 level 区分是否参与上层合计 | "流动资产小计" |

### 3.2 level 规则

- 顶级明细行 `level=1`
- 子项行 `level=2`（属于上一个 level=1 行的拆分）
- 更深层嵌套（如公允价值层次表）按层级递增，但竖加主要按 level=1 的 total 行验算

### 3.3 子项判断（关键，决定竖加是否误报）

子项标记变体无穷，需综合判断：

**显式标记**：
- `[子项]` 标记的行
- "其中："开头的行
- "其中"开头（无冒号）的行

**"其中："的延续规则**（关键）：
"其中："行后面的连续行，直到遇到下一个 level=1 的 detail 行或 total 行，**也都是子项**。

举例：
```
社会保险费                    ← level=1, detail
  其中：医疗保险费             ← level=2, sub_detail
  工伤保险费                   ← level=2, sub_detail（延续）
  生育保险费                   ← level=2, sub_detail（延续）
住房公积金                     ← level=1, detail（子项区间结束）
```

```
交易性金融资产                 ← level=1, detail
  其中：衍生金融工具            ← level=2, sub_detail
  延迟定价安排                  ← level=2, sub_detail（延续）
债权投资                       ← level=1, detail（子项区间结束）
```

**带编号前缀**：
- "组合1："、"组合2：" 紧跟在某个大类后面 → 子项
- "（1）"、"(1)" 紧跟在某个大类后面 → 子项

**判断方法**：如果一行后面紧跟"其中："行，说明该行是大类（level=1），"其中"行及其后续行都是子项（level=2），直到遇到下一个大类行。

### 3.4 "减："/"加："前缀处理（关键）

**"减："/"加："开头的行不是子项**，它们是参与合计的明细行，只是 op 不同：

- "减：" / "减:" 开头 → `op="-"`，`type=detail`，`level=1`
- "加：" / "加:" 开头 → `op="+"`，`type=detail`，`level=1`

举例（利润表格式）：
```
营业收入                       ← op="+", detail
  减：营业成本                 ← op="-", detail（不是子项！）
  税金及附加                   ← op="-", detail
  销售费用                     ← op="-", detail
营业利润                       ← total
```

**代码兜底**：即使 Claude 偶尔把"减："行标成子项，代码侧会强制修正为 `op=-, type=detail, level=1`。

### 3.5 op 规则

- 默认 `op="+"`
- `[减项]` 标记或"减："/"减:"开头 → `op="-"`
- `[加项]` 标记或"加："开头 → `op="+"`
- 负数是原始数据特征（如 `(1,234.56)`），**不影响 op**（op 仍为 "+"，负号已在数值中）

## 四、列标注规则

### 4.1 type 取值

| type | 含义 | 判断依据 |
|---|---|---|
| `label` | 文字标签列（项目名称） | 第一列通常是项目名称 |
| `amount` | 金额列 | 含数字、千分位逗号、括号负数 |
| `percentage` | 百分比列 | 含 `%` 或"比例"/"损失率"/"计提比例" |
| `text` | 其他文本列 | 备注列、性质说明列 |

### 4.2 name 字段

`name` 用列名中最具体的名称。多行表头合并后，列名形如"年末余额-账面余额"，`name` 取完整合并名。

### 4.3 行号列号约定

行号、列号从 0 开始。

---

## 五、横加判断规则

横加是列间的加减关系（如账面余额−坏账准备=账面价值）。Claude 判断是否存在横加关系，代码按判断验算。

### 5.1 常见横加模式

| 模式 | 表达式 | 典型科目 |
|---|---|---|
| 账面余额 − 坏账准备 = 账面价值 | 应收账款、其他应收款、应收票据、合同资产 | 坏账准备可作为单独列 |
| 原价 − 累计折旧 − 减值准备 = 账面价值 | 固定资产 | 折旧和减值分列 |
| 原价 − 累计摊销 − 减值准备 = 账面价值 | 无形资产 | |
| 期初 + 本期增加 − 本期减少 = 期末 | 变动表通用 | 固定资产、应付职工薪酬等 |

### 5.2 不应横加的情况（避免误报）

**年末列与年初列不能混合横加**：
- "年末余额-账面余额"和"年初余额-账面余额"是两个不同时点的列
- operands 和 result_col 必须属于同一时点
- 仔细看列名中的时点前缀

**百分比列不参与金额加减**：
- 损失率、计提比例等百分比列，不能与金额列做加减
- 若 result_col 或任一 operand col 是百分比列，跳过该横加关系

**递延所得税表的税率关系不是加减**：
- "暂时性差异"和"递延所得税资产/负债"之间是 **乘以税率** 的关系，不是简单加减
- 这种表不横加

**只有 operand 列没有 result 列时不横加**：
- 如表格只有"账面余额"和"坏账准备"两列，没有"账面价值"列，不应横加（没有结果列可验算）

**按分类维度分列的表格不横加**：
- 如按账龄分列金额（1 年以内/1-2 年/2-3 年/3 年以上），列间无加减关系

**某列为空的行跳过**：
- 横加验算时，若某行 operand 列或 result 列为空（无数字），跳过该行

### 5.3 横加判断输出格式

```json
{
  "horizontal": [
    {
      "rows": [2, 3, 4],
      "formula": "账面余额 - 坏账准备 = 账面价值",
      "operands": [
        {"col": 1, "op": "+"},
        {"col": 3, "op": "-"}
      ],
      "result_col": 5
    }
  ]
}
```

**operands 规则**：
- 第一个 operand 的 op 通常是 `"+"`（被减数/加数）
- op 为 `"-"` 表示该列要被减去
- 计算方式：按 operands 顺序，op="+" 则加，op="-" 则减，最终结果应等于 result_col 的值
- 列号必须与列标注中的 col 一致
- 无横加则 `horizontal` 为空数组
- **宁可少报不可误报**（误报会制造噪音，漏报可接受）

## 六、标注输出 JSON 格式

Claude 对一张表格的完整标注输出（结构标注 + 横加判断合并）：

```json
{
  "rows": [
    {"row": 0, "level": 1, "type": "detail", "op": "+"},
    {"row": 1, "level": 1, "type": "detail", "op": "+"},
    {"row": 2, "level": 2, "type": "sub_detail", "op": "+"},
    {"row": 3, "level": 1, "type": "detail", "op": "-"},
    {"row": 4, "level": 1, "type": "total", "op": "+"}
  ],
  "columns": [
    {"col": 0, "type": "label", "name": "项目"},
    {"col": 1, "type": "amount", "name": "账面余额"},
    {"col": 2, "type": "amount", "name": "坏账准备"},
    {"col": 3, "type": "amount", "name": "账面价值"}
  ],
  "horizontal": [
    {
      "rows": [0, 1, 3, 4],
      "formula": "账面余额 - 坏账准备 = 账面价值",
      "operands": [
        {"col": 1, "op": "+"},
        {"col": 2, "op": "-"}
      ],
      "result_col": 3
    }
  ]
}
```

**容错**：代码侧用 `json_repair` 解析，即使 Claude 返回的 JSON 有轻微格式问题（多余逗号、单引号等）也能解析。完全无法解析时降级（见第八节）。

## 七、批量标注提效建议

大报告附注可能有几十张表，逐张标注调用成本高。提效策略：

### 7.1 一次提交多张小表

把多张小表（如应收票据、应收账款、其他应收款三张结构相似的表）合并到一次 prompt 调用，Claude 一次返回多张表的标注结果：

```json
{
  "tables": [
    {"table_id": "应收票据", "annotation": {...}},
    {"table_id": "应收账款", "annotation": {...}},
    {"table_id": "其他应收款", "annotation": {...}}
  ]
}
```

**适用条件**：表格行数/列数较小（< 30 行 < 10 列），结构相似。

### 7.2 大表单独处理

行数 > 30 或列数 > 10 的大表（如固定资产变动表、按账龄分列的应收账款明细）单独标注，避免上下文过长导致标注质量下降。

### 7.3 跳过转置表

若表头行中含数字（行列互换的转置表），跳过标注——这类表结构特殊，竖加横加规则不适用。

## 八、标注失败降级策略

Claude 标注可能因以下原因失败：
- JSON 格式严重错误，`json_repair` 也无法解析
- 表格结构过于复杂（深层嵌套、合并单元格混乱）
- 表格内容为空或纯文本（无数字可验算）

### 8.1 降级为仅验证合计行

标注失败时，代码侧降级为：
- **仅验证 total 行**：用代码识别"合计"/"总计"/"小计"等明显合计行，把其上方所有非空数字行（不区分 detail/sub_detail）求和核对
- **不做横加**：横加关系依赖 Claude 判断，标注失败时无法横加，跳过
- **报告中标记**："AI 标注失败"标记该表，提示人工复核

### 8.2 代码兜底修正

即使标注成功，代码侧对以下情况强制修正（AI 偶尔标错）：
- "减："/"减:"开头的行 → 强制 `op=-, type=detail, level=1`
- "加："/"加:"开头的行 → 强制 `op=+, type=detail, level=1`
- "其中"/"其中："/"其中:"开头的行及其后续缩进行 → 强制 `type=sub_detail, level=parent_level+1`

这些兜底规则在 `run_check.py` 的竖加推断逻辑中实现，确保即使 AI 标注有瑕疵也不影响验算确定性。

## 九、Prompt 模板（可直接复制使用）

以下是给 Claude 的标注 prompt，可直接用于附注表格标注。把 `{table_text}` 替换为表格的文本表示（含列名行 + 数据行，行号从 0 开始）。

### 9.1 结构标注 + 横加判断（合并 prompt）

```
你是注册会计师，审阅审计报告附注表格。分析结构，标注每行的层级和类型、每列的数据类型，并判断是否存在横加（列间勾稽）关系。直接返回 JSON。

表格：
{table_text}

【行标注规则】
type 取值：
- "detail": 明细数据行
- "sub_detail": 子项行（属于上一个 detail 行的明细拆分，竖加时排除）
- "total": 合计/总计行
- "subtotal": 小计行（标签必须含"小计"二字）

level 规则：
- 顶级明细行 level=1
- 子项行 level=2（属于某个 detail 行的拆分）

**硬性规则（优先级最高，必须遵守）**：
- 以下行必须标记为 type=sub_detail 且 level=2（竖加验算时必须排除，否则会导致重复计算）：
  * "其中"或"其中："开头的行（无冒号或带冒号都算）
  * "[子项]"标记的行
  * "组合X："、"（X）"、"[X]"等编号前缀的行，且紧跟在某个大类后面
  * 上述行的后续连续行（直到遇到下一个 level=1 的行或合计行），也都是子项
- **明确标记 is_subitem=true**：对于 sub_detail 行，在返回 JSON 中额外添加 "is_subitem": true 字段，便于代码侧快速判断

**减项/加项处理（特殊）**：
- "减：" / "减:" / "[减项]"开头的行 → type=detail, level=1, op="-"（不是子项！）
- "加：" / "加:" / "[加项]"开头的行 → type=detail, level=1, op="+"（不是子项！）

子项判断（关键）：
- [子项] 标记的行是子项
- "其中："开头的行是子项，且它后面连续的行（直到遇到下一个 level=1 的行或合计行）也都是子项
- 带编号前缀的行（如"组合1："、"（1）"）如果紧跟在某个大类后面，也是子项
- "减："/"加："开头的行不是子项，它们是参与合计的明细行，只是 op 不同

op 规则：
- 默认 "+"
- [减项] 标记或"减："/"减:"开头的行 op="-"
- [加项] 标记或"加："开头的行 op="+"
- 负数是原始数据，不影响 op

【列标注规则】
type 取值：label(文字) / amount(金额) / percentage(含%或比例或损失率) / text(其他)

**硬性规则（优先级最高，必须遵守）**：
- 列名含以下关键词之一 → type=percentage（绝对不是 amount），该列不参与竖加横加验算：
  * "%"
  * "比例"
  * "百分比"
  * "rate"
  * "损失率"
  * "计提比例"
  * "占比"
- 上述规则优先于其他判断依据，即使该列含数字，也必须标记为 percentage

name: 用列名中最具体的名称

【横加判断规则】
常见横加模式：
- 账面余额 - 坏账准备/减值准备 = 账面价值
- 期初 + 本期增加 - 本期减少 = 期末
- 原价 - 累计折旧 - 减值准备 = 账面价值

不应横加：
- 年末列与年初列属于不同时点，不能混合横加（operands 和 result_col 必须同一时点）
- **百分比列不参与金额加减**：若 result_col 或任一 operand col 的 type=percentage，跳过该横加关系
- 递延所得税表中"暂时性差异"与"递延所得税资产/负债"是乘税率关系，不横加
- 只有 operand 列没有 result 列时不横加
- 按分类维度分列的表格不横加
- **百分比表跳过横加**：若表头含"%"、"比例"且数值列都在 0-100 范围，整表跳过横加（这是百分比分析表，不是金额表）
- 宁可少报不可误报

返回：
{
  "rows": [{"row": 行号, "level": 层级, "type": "类型", "op": "+"或"-", "is_subitem": true或false}],
  "columns": [{"col": 列号, "type": "类型", "name": "列名"}],
  "horizontal": [
    {"rows": [行号], "formula": "A - B = C", "operands": [{"col": A列号, "op": "+"}, {"col": B列号, "op": "-"}], "result_col": C列号}
  ]
}

**重要**：
- 对于 sub_detail 行，必须设置 "is_subitem": true
- 对于 percentage 类型的列，必须明确标记 type=percentage

operands 规则：
- 第一个 operand 的 op 通常是 "+"（被减数/加数）
- op 为 "-" 表示该列要被减去
- 计算方式：按 operands 顺序，op="+" 则加，op="-" 则减，最终结果应等于 result_col 的值
- 列号必须与上面"列标注"中的 col 一致
- 无横加则 horizontal 为空数组

行号列号从0开始。直接返回 JSON，不要加 markdown 代码块标记。
```
